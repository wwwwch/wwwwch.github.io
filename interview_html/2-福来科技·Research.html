整体框架

一、Transformer 与序列建模基础

1. Transformer 的注意力机制、位置编码、QKV 是怎么实现计算的？

标准回答

Transformer 通过自注意力机制（Self-Attention）、QKV 线性映射和位置编码来建模序列关系。

  • 自注意力机制：让每个 token 与其他 token 建立全局依赖。
  • Q、K、V：把输入映射到“查询、匹配、取值”三个空间。
  • 位置编码：弥补 Attention 本身不具备顺序感知的问题。
Attention(Q, K, V) = softmax((QKᵀ) / √dₖ) V

可以这样理解：Q 表示“我现在想找什么信息”，K 表示“我这里有什么信息可供匹配”，V 表示“真正要被聚合的内容”。模型通过 Q 和 K 的相似度计算注意力权重，再对 V 加权求和，完成一次信息交互。

位置编码告诉模型“内容之外，顺序也重要”。否则模型只知道有哪些 token，却不知道谁在前、谁在后。

高频追问

  • 为什么要除以 √dₖ？
  • 多头注意力的意义是什么？
  • Transformer 为什么比 RNN 更适合长序列？
  • 位置编码为什么必需？

追问回答要点

为什么除以 √dₖ

Q 和 K 点积后，维度越大，数值越容易变大，softmax 会变得过于尖锐，导致训练不稳定。除以 √dₖ 是缩放操作，目的是让梯度更稳定。

多头注意力的意义

不同 head 可以在不同子空间学习不同关系。例如有的 head 关注局部搭配，有的关注长程依赖。本质是提升表达能力和建模多样性。

为什么比 RNN 更适合长序列

RNN 逐步传递信息，长距离依赖容易衰减；Transformer 通过全局交互让任意两个 token 的路径更短，并且支持并行计算。

位置编码为什么必需

Self-Attention 本身对输入顺序不敏感。没有位置编码时，模型无法区分“我打你”和“你打我”这类词集合相同但顺序不同的句子。

二、多模态与训练范式

2. 什么是多模态大模型？

标准回答

多模态大模型是能够统一理解、对齐和生成多种模态信息的模型，例如文本、图像、音频、视频。

它的关键不只是“支持多输入”，而是能把不同模态映射到相对统一的语义空间，再在这个空间里进行推理和生成。

核心理解三层

  • 多输入输出：支持图文问答、语音助手、视频理解等任务。
  • 统一表示：不同模态经过编码后进入共享语义空间。
  • 联合推理：模型不是分别处理文本和图像，而是跨模态融合信息。例如视觉问答中，模型既要理解图片内容，也要理解文本问题，最终生成答案。

因此，多模态大模型不是单模态能力的简单叠加，而是跨模态协同。

高频追问

  • 多模态大模型和传统图文模型的区别是什么？
  • 多模态为什么是大模型的重要方向？
  • 多模态训练的难点有哪些？

追问回答要点

和传统图文模型的区别

传统图文模型很多是任务级拼接，例如图像特征加文本特征后用于分类；多模态大模型更强调统一表征和通用生成能力，泛化能力更强。

为什么是重要方向

真实世界信息天然是多模态的，人类交流也不是纯文本。要做更通用的智能体，必须同时理解语言、视觉、声音等信息。

训练难点

  • 不同模态的数据分布差异大。
  • 不同模态的长度和表示形式不同。
  • 跨模态对齐困难。
  • 高质量标注数据昂贵。
  • 训练和推理成本更高。

三、强化学习

3. 什么是强化学习？

标准回答

强化学习是让智能体通过与环境交互、接收奖励信号、不断调整策略，从而学习最优行为的方法。

它和监督学习最大的不同是：监督学习有明确标签，强化学习只有延迟奖励，模型需要自己探索什么行为更优。

核心要素

AgentEnvironmentStateActionReward

强化学习的目标不是最大化单步收益，而是最大化长期累计回报。因此它本质上解决的是序列决策问题。

例如在游戏、机器人控制、推荐排序中，当前动作往往会影响后续状态，所以强化学习强调长期策略，而不是一次性的局部判断。

高频追问

  • 强化学习和监督学习的本质区别是什么？
  • 什么是延迟奖励问题？
  • 强化学习在大模型里有什么应用？

追问回答要点

和监督学习的区别

监督学习是已知输入和正确输出；强化学习没有明确标准答案，只能通过奖励信号判断行为是否好，核心是试错和策略优化。

延迟奖励问题

当前动作的好坏不一定立刻体现，可能要经过多步后才看到结果。这就是 credit assignment 问题，也就是“功劳如何分配”。

在大模型中的应用

典型应用是 RLHF。先让模型具备基础能力，再利用人类偏好反馈优化输出风格和行为边界，使回答更符合预期。

四、Agent 基础范式

4. 什么是 ReAct？

标准回答

ReAct（Reasoning + Acting）是一种将推理能力和工具调用能力结合起来的 Agent 范式。

它不是让模型一次性直接给答案，而是按照以下循环逐步解决问题：

Thought → Action → Observation → Thought → ...

ReAct 的价值包括两方面：一方面，模型可以通过分步推理提高复杂任务完成率；另一方面，模型可以借助工具补足自身知识和执行能力的边界，例如搜索、数据库查询、调用 API。

可以把 ReAct 理解为：把“只会说”的模型推进为“会思考、会查、会做”的系统。这也是 Agent 与普通对话模型的关键区别之一。

高频追问

  • ReAct 和普通 Chain-of-Thought 的区别是什么？
  • ReAct 为什么能减少幻觉？
  • 哪些任务适合 ReAct？

追问回答要点

和 Chain-of-Thought 的区别

Chain-of-Thought 偏重内部推理；ReAct 不仅推理，还能执行动作，并根据外部反馈修正后续步骤。

为什么能减少幻觉

遇到不确定信息时，模型可以调用工具获取外部证据，而不是只依赖参数记忆硬答。

适合的任务

搜索问答、数据查询、多步决策、任务执行类场景都适合 ReAct，尤其是需要“边查边做”的问题。

五、Ask / Plan / Execute 三种模式

5. Ask / Plan / Execute 三种模式有什么区别？

标准回答

Ask、Plan、Execute 是 Agent 常见的三种交互和执行模式，核心区别在于：是否先澄清、是否先规划、是否直接执行，以及用户参与程度有多高。

模式定义适用场景
Ask先询问再行动。需求不明确、信息不足的场景。
Plan先拆解任务并展示执行方案，再根据情况执行。复杂任务和高可控场景。
Execute目标明确时直接执行。高确定性、低风险任务。

本质上，这是在平衡自动化效率和执行可控性。越靠近 Ask / Plan，越稳；越靠近 Execute，越快。

高频追问

  • 三种模式分别适合什么场景？
  • 为什么很多系统不直接用 Execute？
  • Plan 模式的价值是什么？

追问回答要点

适合场景

Ask 适合需求模糊；Plan 适合复杂任务；Execute 适合简单且定义明确的任务。

为什么不总是 Execute

如果问题理解错了，直接执行的代价会很高，尤其涉及外部操作、数据修改、资源消耗时。

Plan 的价值

Plan 让任务过程更透明，便于用户确认、系统审计和中间纠错，适合企业级场景。

六、多智能体实现

6. 多智能体如何实现？

标准回答

多智能体系统的实现，本质是通过任务拆分、角色分工、通信协作和调度控制，把复杂问题交给多个 Agent 协同完成。

常见实现流程

  1. 先把大任务拆成子任务。
  2. 给不同 Agent 分配角色，例如 Planner、Executor、Reviewer、Router。
  3. 通过共享上下文、消息传递或状态存储进行协作。
  4. 由主控流程或调度器决定执行顺序，可串行也可并行。

多智能体不是“Agent 越多越强”。当任务天然具备可分工性时，多 Agent 能提升模块化、专业化和可扩展性；但前提是调度和状态管理要设计好，否则复杂度会反过来吞噬收益。

高频追问

  • 多智能体相比单智能体的优势和代价是什么？
  • 多智能体通信一般怎么做？
  • 什么情况下不适合多智能体？

追问回答要点

优势和代价

优势是分工明确、可并行、可复用；代价是协调成本高、状态一致性更难、调试复杂度更高。

通信方式

常见通信方式包括共享 memory、消息队列、blackboard、统一状态存储等。本质是让各 Agent 拿到自己需要的上下文。

不适合的情况

当任务本身很简单、链路短、没有明显角色分工时，多智能体反而会增加系统开销。

七、Agent 工程与框架体系

7. LangChain 和 LangGraph 有什么区别？

标准回答

LangChain 和 LangGraph 的关系可以理解为：LangChain 偏向组件化封装，LangGraph 偏向复杂流程编排。

框架定位典型能力
LangChain组件化封装，更像“搭积木”。快速组合 Prompt、模型、检索器、工具调用等，适合快速搭建原型。
LangGraph复杂流程编排，更像“工作流引擎”。图结构、显式状态管理、循环、分支、回溯、多 Agent 协作。

当系统从 Demo 走向真实 Agent 编排时，仅靠线性 Chain 往往不够，这时 LangGraph 的价值会更明显。

高频追问

  • 什么场景更适合 LangGraph？
  • LangChain 是否已经被 LangGraph 替代？
  • LangGraph 的核心优势是什么？

追问回答要点

适合 LangGraph 的场景

需要状态持久化、循环重试、条件分支、多 Agent 编排的复杂流程。

是否被替代

不是。LangChain 仍然适合快速封装基础能力，LangGraph 更像是面向复杂编排的上层能力。

核心优势

显式状态管理和图式流程控制，使复杂 Agent 系统更可控、更可调试。

八、Agent Harness

8. 什么是 Agent Harness？

标准回答

Agent Harness 可以理解为 Agent 的运行、管理、评估和测试框架。

它不直接定义 Agent 的推理逻辑，而是提供统一环境，承载 Agent 的执行过程、工具调用、上下文注入、行为记录和结果评估。

通常包含的层面

  • 执行管理：控制 Agent 的运行流程。
  • 环境封装：管理 memory、工具、上下文。
  • 评测监控：记录轨迹、做回放、做效果分析。

Harness 更接近“运行时 + 测试平台”。如果说 Agent 是业务逻辑，那么 Harness 就是让它可运行、可测量、可优化的工程底座。

高频追问

  • Agent Harness 和普通框架有什么不同？
  • 为什么 Agent 系统需要 Harness？
  • Harness 一般重点评估什么？

追问回答要点

和普通框架的区别

它不只是开发接口，更强调运行控制、日志轨迹、回放复现和评估能力。

为什么需要

Agent 是动态系统，不像固定模型推理那么可预测。没有 Harness 很难定位问题，也很难做系统化评估。

重点评估内容

任务成功率、工具使用正确率、轨迹合理性、成本、时延、稳定性等。

九、Skill 与 MCP

9. Skill 和 MCP 的区别是什么？

标准回答

Skill 和 MCP 不是同一层面的概念。Skill 是能力封装，MCP（Model Context Protocol）是工具接入和上下文交互的协议标准。

概念关注点类比
Skill模型能做什么，例如搜索、计算、数据库查询、文件读取。具体 API 能力。
MCP这些能力怎样被标准化暴露给模型调用，即统一工具接口和上下文交互方式。API 接入规范。

Skill 是功能单元，MCP 是让这些功能单元能被模型一致发现、调用和管理的协议层。

高频追问

  • 为什么需要 MCP 这种协议？
  • Skill 不通过 MCP 能不能用？
  • MCP 对 Agent 工程有什么价值？

追问回答要点

为什么需要协议

工具来源多、接口风格各异，没有统一规范会导致接入成本高、扩展困难。

Skill 不通过 MCP 能不能用

可以，但会更碎片化。MCP 的价值在于标准化、可扩展和跨系统兼容。

对 Agent 工程的价值

MCP 能降低工具接入成本，让 Agent 更容易发现和调用外部能力，也便于治理和维护。

十、主题间的体系化联系

10. 这些问题之间如何形成体系？

标准回答

这些知识点可以分成三层理解：

  • 模型机制层：Transformer、QKV、位置编码，解决模型如何理解和表示信息。
  • 能力扩展层：多模态大模型、强化学习，解决模型如何处理更丰富的输入，以及如何通过奖励优化行为。
  • 系统工程层：ReAct、Ask / Plan / Execute、多智能体、LangChain / LangGraph、Agent Harness、Skill / MCP，解决如何把模型从“会生成”升级为“会调用工具、会执行任务、能被系统化管理”。

换句话说：Transformer 是底层建模基础；多模态和强化学习扩展能力边界；Agent 框架和协议让这些能力真正落到可执行系统里。

高频追问

  • 现在技术演进的主线是什么？
  • 为什么单有大模型还不够？
  • 面试时如何把这些知识点串起来表达？

追问回答要点

技术演进主线

从“单任务模型”到“统一大模型”，再到“结合工具调用、检索、工作流和评估体系的智能系统”。

为什么单有大模型还不够

因为参数记忆有限、执行能力有限、可靠性有限。真实业务需要外部知识、工具执行和工程治理。

如何串起来表达

可以按“模型原理 → 能力扩展 → 系统落地”三层结构讲，体现体系化理解，而不是孤立背术语。

十一、适合面试收尾的总结表达

11. 如果面试官让你整体概括，你可以怎么回答？

标准回答

可以这样概括：这些知识点可以统一放在“从模型到系统”的演进路径里看。Transformer 解决了序列建模的核心问题，是现代大模型的基础；多模态和强化学习扩展了模型的理解和优化能力；而 ReAct、多智能体、LangGraph、MCP 这类概念，本质上是在解决大模型如何真正接入外部世界、执行复杂任务并被工程化管理的问题。

所以，这些面试问题看起来分散，但其实是同一条主线：从“模型会生成”走向“系统会理解、会调用、会执行、可评估、可落地”。

高频追问

  • 你觉得未来重点会落在哪一层？
  • 你更看重模型能力还是系统能力？

追问回答要点

未来重点落在哪一层

未来重点会更多落在系统能力层。模型能力还会持续提升，但真正决定业务价值的是可靠性、成本、工具集成和执行闭环。

更看重模型还是系统

两者都重要，但从落地角度看，更应重视系统能力。因为模型再强，如果不能接工具、不能控流程、不能评估，就很难稳定进入生产环境。

面试表达建议：不要把概念孤立背诵。推荐按“模型原理 → 能力扩展 → 系统落地 → 工程治理”的逻辑串联，形成体系化表达。
